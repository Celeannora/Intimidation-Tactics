# Database Schema

> Engine: [Dexie.js](https://dexie.org/) v3 (IndexedDB wrapper)  
> Database name: `mtgDeckBuilderDB`  
> Current schema version: **3**

---

## Overview

All persistent state lives in a single Dexie database with four object stores. No data ever leaves the device unless the user explicitly copies their deck to an external target. This is the core pillar of the offline-first architecture (see [ADR-001](./adr/001-offline-first-dexie.md)).

## Object Stores

**Multi-value indices** (`*fieldName`): `keywordsJson`, `colorsJson`, `colorIdentityJson` — Dexie parses these JSON arrays so each element becomes a separate index entry, enabling `db.cards.where("colorsJson").equals("U")` style queries.

**Typical row count:** 30,000–35,000 records after a full Scryfall bulk-data import.

### `meta`

Primary key: `key` (string)

| Key | Value | Description |
|-----|-------|-------------|
| `lastImportedAt` | ISO timestamp string | When the last successful bulk import ran |
| `cardCount` | numeric string | Number of card records in `cards` store |

Populated atomically with the `cards` store inside a `replaceAllCards` transaction to prevent partial-state reads.

---

### `savedDecks`

Primary key: `id` (string, generated by `makeId()`)

| Field | Type | Indexed | Notes |
|-------|------|---------|-------|
| `id` | `string` | PK | Nanoid/UUID deck identifier |
| `name` | `string` | – | User-assigned deck name |
| `updatedAt` | `number` | ✓ | `Date.now()` — used for sort-by-recent |
| `mainboard` | `Record<string, number>` | – | `oracleId → quantity` map |
| `sideboard` | `Record<string, number>` | – | `oracleId → quantity` map |
| `wins` | `number` | – | Match record |
| `losses` | `number` | – | Match record |
| `draws` | `number` | – | Match record |
| `pins?` | `Record<string, number>` | – | Pinned cards immune to optimizer; optional for backward compat |

The `mainboard`/`sideboard`/`pins` fields are stored as plain JS objects (Dexie serialises them as JSON). Card details are joined at read-time via `oracleId` against the `cards` store — the deck store only holds oracle IDs, not full card data.

### `matchResults`

Primary key: `id` (auto-increment integer)

| Field | Type | Indexed | Notes |
|-------|------|---------|-------|
| `id` | `number` | PK (auto) | Auto-incremented |
| `deckId` | `string` | ✓ | FK → `savedDecks.id` |
| `opponent` | `string` | – | Free-text opponent archetype/name |
| `result` | `"win" \| "loss" \| "draw"` | – | Match outcome |
| `notes` | `string` | – | Free-form notes |
| `playedAt` | `number` | ✓ | `Date.now()` — enables timeline queries |

## Key Operations

### Bulk Card Import

```ts
// src/lib/db.ts
export async function replaceAllCards(cards: CardRecord[], importedAt: string) {
  await db.transaction("rw", db.cards, db.meta, async () => {
    await db.cards.clear();
    await db.cards.bulkPut(cards);
    await db.meta.bulkPut([
      { key: "lastImportedAt", value: importedAt },
      { key: "cardCount",      value: String(cards.length) },
    ]);
  });
}
```

This is the only write path for card data. The `rw` transaction ensures atomicity — either the full set lands, or the old set is preserved.

### Deck Persistence

Decks are saved via `db.savedDecks.put(deck)` and loaded by `id`. The Zustand store (`src/store/`) holds the in-memory working copy; changes are flushed to Dexie only on explicit save actions.

## Storage Budget Considerations

| Store | Typical Size |
|-------|-------------|
| `cards` (35k rows × ~800 bytes) | ~28 MB |
| `savedDecks` | < 1 MB (text-only) |
| `matchResults` | < 1 MB |
| **Total** | **~29 MB** |

Browser IndexedDB quotas vary by platform (typically 50–500 MB or 10–20% of available disk). A first-run warning is shown if the import would exceed 80% of the estimated quota. See `PWA_STRATEGY.md` for quota management details.

## Upgrade Safety Rules

1. **Never remove an index** in a version upgrade without bumping the version number.
2. **Never change a primary key** after data exists; create a new store instead.
3. All `version(N).stores()` calls are cumulative — omit a store name only if you intentionally want to delete it.
4. Test migrations against a populated v1 DB before shipping a version bump.
